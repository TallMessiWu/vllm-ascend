1背景与两条复现命令

用户反馈：Qwen3.5-35B-A3B 在 TP1 时精度与性能都正常；切到 TP2 后出现精度问题，且 MoeInitRouting 算子耗时劣化严重。 但两条实跑命令的关键参数并不一致：

用途权重TPDPenable-expert-parallel量化
单卡基准（正常）Qwen3.5-35B-A3B-w8a8-mxfp11否mxfp 量化
双卡 profiling（看性能）Qwen3.5-35B-A3B21否非量化 bf16
双卡（看精度）Qwen3.5-35B-A3B21是非量化 bf16
⚠ 对照里同时变了 3 个变量

并行度（TP 1→2）、模型/量化（mxfp→bf16）、EP（性能不开 / 精度开）。 三个变量耦合在一起，不能把所有差异都记到"TP2"头上。下面把性能与精度两条线分开归因。

2代码路径：TP/EP 如何改变 init_routing 的入参

MoE 在 vllm-ascend 里的通信方式由 select_moe_comm_method 决定。关键判断（vllm_ascend/ascend_forward_context.py:270）：

if not vllm_config.parallel_config.enable_expert_parallel or get_ep_group().world_size == 1:
    moe_comm_type = MoECommType.ALLGATHER   # 不开 EP → 一律走 AllGather

AllGather 路径下，TokenDispatcherWithAllGather.token_dispatch 给 init_routing 准备 active_expert_range（vllm_ascend/ops/fused_moe/token_dispatcher.py:383-403）：

if expert_map is not None:                 # 开 EP（ep_size>1）才有 expert_map
    mask = expert_map[topk_ids] != -1
    topk_weights = topk_weights * mask        # 非本地专家权重置 0
    first_expert_idx = ep_rank * num_experts_local
    last_expert_idx  = first_expert_idx + num_experts_local
else:                                         # 不开 EP（含纯 TP）
    first_expert_idx = 0
    last_expert_idx  = num_experts_local      # = 全部专家 E

DeviceOperator.npu_moe_init_routing(
    ..., active_expert_range=[first_expert_idx, last_expert_idx], quant_mode=quant_mode)
不开 EP（TP1 或纯 TP2）：active_expert_range = [0, E] 全集
开 EP（精度命令） ：active_expert_range = [0, E/2] 或 [E/2, E] 子区间 + topk_weights*mask

而具体落到哪个底层算子，取决于设备（vllm_ascend/device/device_op.py）：

  • 非 A5：torch.ops._C_ascend.npu_moe_init_routing_custom（本 fork 的自定义算子）
  • A5：torch_npu.npu_moe_init_routing_v2（CANN 内置）

3问题 A — MoeInitRouting 性能劣化（profiling 命令没开 EP）

关键观察

不开 EP 时，expert_map = None，TP=2 的 active_expert_range 仍是 [0, E]。 也就是说 TP 翻倍并不改变 full-load 判定。真正在两条命令间变化的是 quant_mode（量化模型 ≠ bf16）。

3.1 full-load 高效分支 vs 逐 token 慢分支

自定义算子 host tiling 的 IsFullLoad()（csrc/moe/moe_init_routing_custom/op_host/moe_init_routing_custom_tiling.cpp:644）：

bool IsFullLoad() {
  if (expertStart_ == 0 && expertEnd_ == expertNum_) {   // active_expert_range == 全集
      ep_ = 0;
      perCoreTokens = n_ / aivNum;          // ✅ 每核一次批量处理 n/核数 个 token
  } else {                                  // 子区间（EP 切分后）
      ep_ = 1;
      perCoreTokens = 1;                    // ❗每核每次只处理 1 个 token —— 严重退化
  }
  if (totalLength_ > sortLoopMaxElement || dropPadMode_ == 1) return false;
  ...
  return remainUb > 0;                       // UB 放不下也会退出 full-load
}

注：上面 ep_=0/1 的分叉只在"开 EP 让区间变子集"时才触发；性能 profiling 没开 EP，不走这一支。性能问题的关键在下面的 UB 判定。

3.2 量化 vs bf16：UB 占用差异巨大（性能劣化的真正嫌疑）

同一函数末尾的 UB 预算（第 675–687 行）。quant_mode 取值见 token_dispatcher.py:371-374：量化模型 = 1（int8）或 3（mxfp），bf16 非量化 = −1。

if (quantMode_ == -1) {                    // bf16 非量化
    remainUb -= gatherSpace;             // = cols * 2 * perCoreTokens ← 随 token 数线性增长
} else if (quantMode_ == 0) {             // 静态量化
    remainUb -= gatherSpace + cols*11;
} else {                                 // 动态量化 quant_mode >= 1（含 mxfp=3 / int8=1）
    remainUb -= cols * 13;               // 固定，不乘 perCoreTokens
}
return remainUb > 0;

其中 full-load 时 perCoreTokens ≈ n_ / aivNum（n_ = 本次 token 数）。常量：DYNAMIC_QUANT_FULLLOAD_COLS_BUFFER = 13、STATIC = 11。

推断：bf16 + 大 batch 撑爆 UB → 退出 full-load
  • mxfp 量化模型（基准）：UB 只减一个固定的 cols*13，与 token 数无关 → prefill 大 batch 也能保持 full-load → 快。
  • bf16 非量化模型（profiling）：UB 要减 cols*2*perCoreTokens，--max-num-batched-tokens 65000 这种大 batch 下 perCoreTokens 很大 → remainUb < 0 → IsFullLoad() = false → 退到慢路径 → MoeInitRouting 劣化。

→ 这个劣化由"换成非量化 bf16 模型 + 大 batch"解释，不需要 TP2 参与。单卡若也跑 bf16 非量化，预计一样劣化。

另外，sortLoopMaxElement = (ubSize − aivNum*32)/32/SORT32*SORT32（第 809 行，与量化无关）；当 totalLength_ = n_*k_ 超过它时，无论量化都会退出 full-load。但基准 mxfp 同样是大 batch 却不劣化，说明决定性差异更可能在上面的 gatherSpace 这条 bf16 专属路径上。

4问题 B — 精度问题（命令开了 EP）

这条命令加了 --enable-expert-parallel → ep_size = 2 → expert_map 非 None → active_expert_range 变子区间。此时同时发生：

  • init_routing 走 ep_=1 分支（与 full-load 是两套不同 kernel，本 fork 较新接入、仍在迭代）；
  • dispatch 端 topk_weights *= (expert_map[topk_ids] != -1) 把外部专家权重置 0；
  • combine 端用 torch.abs(expanded_row_idx) 还原顺序（token_dispatcher.py:419-424）。
精度首要嫌疑

子区间 / droppad 场景下 expanded_row_idx 的符号语义、expert_tokens 计数，或 custom/v2 的 ep_=1 分支本身。 这条线确实与 TP/EP 相关，是 TP2 专属问题，与问题 A 是两码事。

7预测与修复方向

问题预测根因方向
A. 性能劣化 非量化 bf16 + 大 batch 使 init_routing 退出 full-load（UB 不足）；非 TP 所致 ① 先做对等对照确认；② 若线上就是要跑 bf16，可考虑大 batch 下的算子侧 full-load UB 优化，或直接用 mxfp 量化权重部署；③ 纯 TP 无需 EP，避免引入子区间路径
B. 精度问题 开 EP → 子区间 ep_=1 分支 / mask + abs(expanded_row_idx) combine 逻辑 ① 加 combine 端张量级日志（expert_tokens 分布 / expanded_row_idx 负值占比）定位；② 临时改走 MC2 / All2All dispatch 规避 init_routing 子区间路径；③ 若锁定算子 ep 分支，算子侧修复

8不确定性 / 待证伪点

  • 设备代次：模型名含 mxfp 强烈暗示 A5；A5 走 npu_moe_init_routing_v2 而非本 fork 的 custom 算子，因此第 3 节的 tiling 常量是同族算子的合理类比而非直接证据。device 字段会一锤定音。
  • profiling 关注阶段：若 profiling 主要看 decode（token 少），bf16 也可能维持 full-load，则劣化另有来源（如通信/all-reduce），需结合 num_tokens 与 elapsed_ms 复核。
  • 精度 root cause："mask 逻辑" vs "算子 ep 分支 bug" 尚未区分，需第 7 节 B 的张量级日志或 TP1/TP2 中间张量对拍。
